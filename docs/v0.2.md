# Structural Precedence Ledger v0.2

## Canonical Structure Registry

## Status

Active specification.

## Purpose

v0.2 introduces a canonical registry for assigning stable
identifiers to structural ideas.

v0.1 records the precedence evidence of one structure.

v0.2 registers multiple structures under a common namespace
and prevents accidental duplication or identity collision.

## Core relationship

```text
Structural Precedence Record
            ↓
Canonical Structure Registry
            ↓
Stable Structure ID
            ↓
Discovery and Relationship Graph
Canonical structure identifier

Every registered structure must receive an identifier using:

STR-YYYY-NNNN

Example:

STR-2026-0001

The identifier must remain stable even when:

the structure name changes;
aliases are added;
the repository moves;
the structure is deprecated;
a successor structure appears.

A removed identifier must not be silently reused.

Required registry entry fields

Each registry entry records:

canonical structure ID;
slug;
English canonical name;
optional Japanese canonical name;
localized aliases;
summary;
originator identity;
precedence record ID;
precedence record URI;
classification;
discovery keywords;
lifecycle state;
supersession relationships;
optional royalty route.
Canonical name rules

A structure has one canonical English name.

It may also have:

one Japanese canonical name;
multiple localized aliases.

Canonical names and aliases must not collide after Unicode
normalization and case folding.

For example, variants caused only by capitalization,
full-width characters, or punctuation may be treated as
the same discovery label.

Origin binding

Each structure is bound to one precedence record through:

origin_binding:
  originator_id: "shidenkai-alpha"
  precedence_record_id: "SPL-0001"
  precedence_record_uri: "https://example.org/records/SPL-0001.yaml"

The same originator and precedence record combination must not
be registered under multiple structure IDs.

Structure types

v0.2 defines the following initial structure types:

protocol
architecture
governance-model
economic-model
philosophical-framework
registry
audit-model
other
Lifecycle states
active

The structure is current and available for use.

deprecated

The structure remains identifiable but is no longer recommended.

superseded

A newer registered structure has replaced it.

A superseded structure must identify at least one successor.

withdrawn

The originator or registry maintainer has withdrawn the structure.

The record and identifier should remain visible for historical
integrity.

Supersession relationships

Two reciprocal fields are used:

supersedes:
  - "STR-2026-0001"

superseded_by:
  - "STR-2026-0002"

If structure B says that it supersedes structure A,
structure A must say that it is superseded by structure B.

One-sided supersession claims fail semantic validation.

Related structures

related_structures records meaningful neighboring structures.

A related structure must:

exist in the same registry;
use a valid canonical structure ID;
not reference itself.

Related structures do not imply derivation or influence.

Semantic validation

JSON Schema validates the shape of the registry.

The Python validator additionally checks:

duplicate structure IDs;
duplicate slugs;
duplicate origin bindings;
canonical name collisions;
alias collisions;
identifier sorting;
unknown related structures;
self-references;
unknown supersession targets;
missing reciprocal supersession links;
superseded structures without a successor.
Non-goals

v0.2 does not:

determine legal ownership;
prove influence;
calculate royalty percentages;
automatically find all aliases;
merge disputed structures;
replace the v0.1 precedence record;
create a derivative graph.

Derivative and convergence graphing is reserved for v0.3.

Core principle

A structure may have many names,
but it must have one stable identity.
